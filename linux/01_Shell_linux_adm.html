1.1 Terminal vs Console¶

Terminal emulator program¶

Also called Terminal window, allows get to a shell from the GUI. Use Ctrl+Alt+T.

Virtual consoles¶

Virtual consoles (tty, or "teletypewriter") are a way to have multiple shell sessions open at once in addition to the graphical interface you are using.

  • Ctrl Alt + F1 through F8 in Debian.
    • GUI is on the Ctrl+Alt+F7 in Debian and Ctrl+Alt+F1 in Fedora,
    • other virtual consoles are typically text-based virtual consoles.

You can use it for debugging or running simultaneous processes.

In the context of Unix and Unix-like operating systems, tty stands for "teletypewriter." Originally, teletypewriters were electromechanical typewriters that could communicate over telegraph lines and were used for early text input and output devices in computing.

In modern usage, tty refers to a terminal device. Here are some key points about tty:

  • Terminal Device: A tty is a terminal device that represents a text input/output environment. This can be a physical terminal, a console, or a virtual terminal.

  • Command: The tty command in Unix-like systems is used to print the file name of the terminal connected to standard input. For example:

  • Types of tty Devices:
    • Physical TTYs: These refer to physical terminals directly connected to the computer.
    • Virtual TTYs (PTYs): These are virtual terminals, such as terminal emulator windows in graphical environments (e.g., xterm, gnome-terminal) or remote sessions (e.g., SSH connections).

Examples of tty Devices

  • /dev/tty: Represents the controlling terminal for the current process.

Historical Context

The term "teletypewriter" comes from the historical use of electromechanical devices to input and output text. These devices were precursors to modern computer terminals and influenced the design and terminology used in Unix systems.

Practical Usage

Understanding tty devices and commands is useful for system administrators and users working with terminal-based interfaces, remote sessions, and script automation. For example, checking the tty can help diagnose issues with terminal sessions or configure terminal settings appropriately.

1.2 Different shells¶

  • sh - one of the earliest UNIX shells, named after its creator Stephen Bourne
  • bash - "Bourne Again Shell", most Linux systems
  • dash - Debian/Ubuntu default scripts' shell, light version of bash
  • ksh - "Korn shell", UNIX System V
  • csh - Sun Microsystems and other BSD
  • tcsh - an improved C shell
  • ash - another Bourne shell look-alike
  • fish

Bash includes features originally developed for sh and ksh shells in early UNIX systems, as well as some csh features. Expect bash to be the default login shell in most Linux systems that you are using, with the exception of some specialized Linux systems (such as some that run on embedded devices) that may require a smaller shell that needs less memory and requires fewer features.

1.5 Commands order¶

Not all of the commands you run are located in directories in your PATH variable. Some commands are built into the shell. Other commands can be overridden by creating aliases that define any commands and options that you want the command to run. There are also ways of defining a function that consists of a stored series of commands.

Here is the order in which the shell checks for the commands you type:

  1. Aliases. These are names set by the alias command that represent a particular command and a set of options. Type alias to see what aliases are set. Often, aliases enable you to define a short name for a long, complicated command.

  2. Shell reserved word. These are words reserved by the shell for special use. Many of these are words that you would use in programming-type functions, such as do, while, case, and else.

  3. Function. This is a set of commands that is executed together within the current shell.

  4. Built-in command. This is a command built into the shell. As a result, there is no representation of the command in the filesystem. Some of the most common commands that you will use are shell built-in commands, such as cd, echo, etc.

  5. Filesystem command. This command is stored in and executed from the computer’s filesystem. (These are the commands that are indicated by the value of the PATH variable.)

Command-line editing¶

By default, the bash shell uses command-line editing that is based on the emacs text editor.

If you prefer the vi command for editing shell command lines, you can easily make that happen. Add the following line to the .bashrc file in your home directory:

set -o vi

The next time you open a shell, you can use vi commands to edit your command lines (see Vim Tutorial for commands).

Command-line completion¶

Here are some of the values you can type partially from a bash shell using Tab key:

  • Command, alias, or function
    • If the text you type begins with regular characters, the shell tries to complete the text with a command, alias, or function name.

VR:
Your own scripts in PATH will not be autocompleted unless you have added this line to the /usr/share/bash-completion/bash_completion file:

complete -c command . source

This command autocompletes the .sh files after the . and source commands.

There is one big disadvantage of this method: you won't be able to autocomplete scripts not in your PATH, you will have to type them manually.

  • Variable

    • If the text you type begins with a dollar sign ($), the shell completes the text with a variable from the current shell.

  • Username

    • If the text you type begins with a tilde (~), the shell completes the text with a username. As a result, ~username indicates the home directory of the named user.

  • Hostname

    • If the text you type begins with the at symbol (@), the shell completes the text with a hostname taken from the /etc/hosts file (Not true for Debian 12. - VR),
    • To add hostnames from an additional file, you can set the HOSTFILE variable to the name of that file. The file must be in the same format as /etc/hosts.
    • This feature rarely works on modern systems since they tend to use DHCP rather than listing host names in the /etc/hosts file (see Shotts).

Pipes are an excellent illustration of how UNIX, the predecessor of Linux, was created as an operating system made up of building blocks. A standard practice in UNIX was to connect utilities in different ways to get different jobs done. For example, before the days of graphical word processors, users created plain-text files that included macros to indicate formatting. To see how the document really appeared, they would use a command such as the following:

$ gunzip < /usr/share/man/man1/grep.1.gz | nroff -c -man | less

In this example, the contents of the grep man page (grep.1.gz) are directed to the gunzip command to be unzipped. The output from gunzip is piped to the nroff command to format the man page using the manual macro (-man). To display the output, it is piped to the less command. Because the file being displayed is in plain text, you could have substituted any number of options to work with the text before displaying it. You could sort the contents, change or delete some of the content, or bring in text from other documents. The key is that, instead of all of those features being in one program, you get results from piping and redirecting input and output between multiple commands.

Sequential commands ;¶

  • one command completing before the next command begins:
$ date ; troff -me verylargedocument | lpr ; date

In this example, I was formatting a huge document and wanted to know how long it would take. The first command (date) showed the date and time before the formatting started. The troff command formatted the document and then piped the output to the printer. When the formatting was finished, the date and time were printed again (so I knew how long the troff command took to complete).

Another useful command to add to the end of a long command line is mail. You could add the following to the end of a command line:

; mail -s "Finished the long command" chris@example.com

Then, for example, a mail message is sent to the user you choose after the command completes.

Background commands &¶

$ troff -me verylargedocument | lpr &

Don’t close the shell until the process is completed or that kills the process. Other ways to manage background and foreground processes are described in Chapter 6, “Managing Running Processes.”

Expanding commands $¶

  • the standard output of a command becomes an argument for another command.

The two forms of command substitution are

  • (command) and
  • `command` (backticks, not single quotes).

The command in this case can include options, metacharacters, and arguments.

$ vi $(find /home | grep xyzzy)

In this example, the command substitution is done before the vi command is run. First, the find command starts at the /home directory and prints out all of the files and directories below that point in the filesystem. The output is piped to the grep command, which filters out all files except for those that include the string xyzzy in the filename. Finally, the vi command opens all filenames for editing (one at a time) that include xyzzy.

This particular example is useful if you want to edit a file for which you know the name but not the location. As long as the string is uncommon, you can find and open every instance of a filename existing beneath a point you choose in the filesystem. (In other words, don’t use grep from the root filesystem or you’ll match and try to edit several thousand files.)

Expanding arithmetic expressions $¶

There are two forms that you can use to expand an arithmetic expression and pass it to the shell:

  • $[expression] or
  • $(expression).

The following is an example:

In [15]:
current_year=$(date +%Y)

echo "Today is $current_year year."
echo "And I am $[$current_year - 1988] years old."

Today is 2024 year.
And I am 36 years old.

$ echo "I am $[2020 - 1957] years old."

I am 63 years old.

The shell interprets the arithmetic expression first (2020 - 1957) and then passes that information to the echo command. The echo command displays the text with the results of the arithmetic (63) inserted.

Here’s an example of the other form:

In [8]:
echo "There are $(ls | wc -l) files in this directory"

There are 12 files in this directory

This lists the contents of the current directory (ls) and runs the word count command to count the number of files (lines in our case) found (wc -l). The resulting number (12, in this case) is echoed back with the rest of the sentence shown.

Expanding variables¶

Variables that store information within the shell can be expanded using the dollar sign $ metacharacter. When you expand an environment variable on a command line, the value of the variable is printed instead of the variable name itself, as follows:

In [11]:
ls -lh $BASH

-rwxr-xr-x 1 root root 1.3M Apr 24  2023 /usr/bin/bash

Using $BASH as an argument to ls -lh causes a long listing of the bash command to be printed.

PROMPT_COMMAND¶






















The PROMPT_COMMAND mechanism is implemented at the shell level. When Bash is about to display a new prompt, it checks the value of the PROMPT_COMMAND variable. If this variable is set and contains a command, Bash executes that command before displaying the prompt.


Internally, Bash has a loop that continuously reads and executes commands. When it encounters the PROMPT_COMMAND variable during this loop, it executes the command stored in PROMPT_COMMAND just before displaying the new prompt.


Here's a simplified overview of how it might work programmatically:


    

  1. Bash reads a command or reaches the end of its loop.
    2. 

  2. Before displaying the prompt, it checks the PROMPT_COMMAND variable.
    3. 

  3. If PROMPT_COMMAND is set and contains a command, Bash executes it.
    4. 

  4. After executing PROMPT_COMMAND, Bash displays the new prompt.
    5. 



This mechanism allows you to execute custom commands or scripts before each new prompt is displayed, providing flexibility and automation capabilities.






















TMOUT¶






















The TMOUT environment variable in Bash controls the auto-logout feature for an idle shell session. When set, it specifies the number of seconds the shell will wait for input before automatically logging out the user.


Let's say you want to set a timeout of 300 seconds (5 minutes). You can set the TMOUT variable like this:


export TMOUT=300



Now, after 5 minutes of inactivity in the shell session, the shell will automatically log you out.


For the TMOUT variable to be effective, it needs to be set in the login shell. This ensures that the timeout applies to the entire session and not just the current subshell.


You can add the export TMOUT=300 line to your .bashrc file to make it persistent across sessions:


echo 'export TMOUT=300' >> ~/.bashrc



After adding this line, the TMOUT variable will be set each time you start a new shell session, ensuring the auto-logout feature is active.




Note: This feature is particularly useful for security purposes, especially on systems where you want to minimize the risk of unauthorized access to unattended or idle shell sessions.


























Creating and using aliases¶






















alias - create a shortcut to any command and options that you want to run later.


$ alias p='pwd ; ls -1'
$ alias rm='rm -i'



In the first example, the letter p is assigned to run the command pwd and then to run ls -1 to print the current working directory and list its contents in column form.


The second example runs the rm command with the -i option each time you type rm. (This is an alias that is often set automatically for the root user. Instead of just removing files, you are prompted for each individual file removal. This prevents you from automatically removing all of the files in a directory by mistakenly typing something such as rm *.)


While you are in the shell, you can check which aliases are set by typing the alias command. If you want to remove an alias, use unalias. (Remember that if the alias is set in a configuration file, it will be set again when you open another shell.)

Setting the prompt¶






















Bash Prompt HOWTO


To make the working directory representation shorter, you can customize the prompt (PS1) variable in your shell configuration file, such as ~/.bashrc or ~/.bash_profile.


You can set the PS1 variable to a shorter format that only displays the last directory in the path. Here's an example of how you can modify the PS1 variable to achieve this:


vi ~/.bashrc

# Set a colored prompt with short working directory format
PS1='\[\033[01;32m\]\u@\h\[\033[00m\]:\[\033[01;34m\]\W\[\033[00m\]\☭ '
:wq

source ~/.bashrc



In this format:


    

  • \[\033[01;32m\] sets the color for the username and hostname to bold green.
    • 

  • \[\033[00m\] resets the color to the default.
    • 

  • \[\033[01;34m\] sets the color for the working directory to bold blue.
    • 

  • \W represents the last part of the current working directory path.
    • 

  • \$ represents the prompt character ($ for regular user, # for root).
    • 



After modifying your ~/.bashrc or ~/.bash_profile file with the updated PS1 variable, reload the configuration in your current terminal session or start a new terminal session for the changes to take effect.


Once you've set the PS1 variable with color codes included for all components, your prompt in Konsole should display the desired colors for the username, hostname, and working directory, along with the shorter format for the working directory. Adjust the color codes and prompt format as needed to suit your preferences.






















PS1 variable¶






















The PS1 (Prompt String 1) variable in Unix-like operating systems, including Linux, defines the format of the primary command prompt. It allows users to customize the appearance of their command prompt by defining various escape sequences and special characters.


In Unix-like operating systems, there are several prompt strings similar to PS1 that serve different purposes:


    

  • PS1 (Prompt String 1): This is the primary command prompt displayed in the terminal. It typically shows information such as the username, hostname, current working directory, and a prompt character.
    • 

  • PS2 (Prompt String 2): This is the secondary prompt displayed when the shell expects more input to complete a command. For example, when you enter a multi-line command, the PS2 prompt is displayed to indicate that the command is not yet complete.
    • 

  • PS3 (Prompt String 3): This prompt is used in shell scripts when the select construct is used to create interactive menus.
    • 

  • PS4 (Prompt String 4): This prompt is used for debugging purposes and is displayed before each command in a script when the set -x option is enabled. It is often used to trace the execution flow of a script.
    • 

  • PROMPT_COMMAND: Runs just before the PS1.
    • 



These prompt strings allow users to customize the appearance and behavior of their shell prompt in various contexts. Each prompt string can be customized using escape sequences and special characters to display information or provide visual cues to the user.


Here are some common components that can be included in the PS1 variable:


    

  • \u: Inserts the username of the current user.
    • 

  • \h: Inserts the hostname up to the first '.' of the system.
    • 

  • \W: Inserts the current working directory (shortened to the last part of the path).
    • 

  • \w: Inserts the full current working directory.
    • 

  • \t: Inserts the current time in 24-hour HH:MM:SS format.
    • 

  • \d: Inserts the date in "Weekday Month Date" format.
    • 

  • \n: Inserts a newline character.
    • 

  • \: Inserts a backslash.

Add scripts to PATH¶






















If you want to add your own commands or shell scripts, place them in the /home/user/bin directory. This directory is automatically added to your path in some Linux systems, although you may need to create that directory or add it to your PATH on other Linux systems. So, as long as you add the command to your bin with execute permission, you can begin using it by simply typing the command name at your shell prompt. To make commands available to all users, add them to /usr/local/bin.




Note: For autocomplete, see Command-line completion.




Unlike some other operating systems, Linux does not, by default, check the current directory for an executable before searching the path. It immediately begins searching the path, and executables in the current directory are run only if they are in the PATH variable or you give their absolute (such as /home/chris/scriptx.sh) or relative (for example, ./scriptx.sh) location.


The path directory order is important. Directories are checked from left to right. So, in this example, if there is a command called foo located in both the /usr/bin and /bin directories, the one in /usr/bin is executed. To have the other foo command run, you either type the full path to the command or change your PATH variable.






















Change for entire system¶






















To permanently update the PATH for all users on Linux, you can add the following line to the /etc/profile file:


PATH="$PATH:/new/dir"



Editing the /etc/profile file requires root privileges. After you've added this line to the file, it will take effect for all users upon their next login. Also see /etc/bash.bashrc file.

		   printf "%s: command not found\n" "$1" >&2
		   return 127
		fi
	}
fi
