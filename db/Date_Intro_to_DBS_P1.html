2. Database System Architecture

2.8 The database management system

Overall purpose of the DBMS is to provide the user interface to the database system, i.e.to support the creation and execution of applications. That is, this is the external level.

Functions of the DBMS

  • Data definition
    • DDL processor/compiler for of the various DDLs to be able to accept data definitions (external schemas, the conceptual schema, the internal schema, and all associated mappings);
  • Data manipulation
    • DML proceccor/compiler to handle requests to retrieve, update, or delete existing data in the db or add new data to the db
    • DML requests can be plannded (operational or production applications) or unplanned (decision support applications);
  • Optimization and execution (see chapter 18 for details)
    • optimizer component implements the optimized DML request, and the run-time manager executes it;
  • Data security and integrity
    • can be carried at compile time or run time, or some mixture of two;
  • Data recovery and concurency (see Part IV)
    • transaction manager (is not usually the part of the DBMS per se) or TP monitor
  • Catalog (also: data dictionary, data repository, data encyclopedia)
    • metadata or descriptors - definitions of other objects in the system, instead of just "raw data" (all the various schemas and mappings are kept in the data dictionary)
    • it is possible to query this database
  • Performance.

Major DBMS functions and components

File manager (File server)

File management system is the component of the underlying operating system that manages stored files, i.e. the DBMS is built on top of file manager. Thus, the user can of a file management system is able to create and destroy stored files and perform simple retrieval and upate operations on stored records in such files.

File managers lack all the benefits of the databases, like data integrity and security, data recovery and concurrncy, and so on.

2.9 Data communications

Data communications (DC) manager is not part of the DBMS but it is an autonomous system in its own right.

The DBMS looks after the database and the DC manager handles all the messages to and from applications that use the DBMS.

2.10 Client/Server Architecture

The basis for distributed processing - the distinct machines can be connected into some kind of communications network in such a way that a single data processing task can be spread across several machines in the network.

  • Server
    • is just the DBMS itself
  • Client
    • are various applications that run on top of the DBMS - both user written applications and built-in (vendor-provided) applications (so called tools).

Tools are designed to assist in the creation and execution of other applications. They can be divided into several more or less distinct classes:

  • query language processors
  • report writers
  • business graphics subsystems
  • spreadshits
  • natural language processors
  • statistical packages
  • copy management or "data extract" tools
  • application generators (including 4GL)
  • other application development tools, including computer-aided software engineering (CASE) products
  • data mining and vusualization tools
    and many others.

These tools are the major factor in "the database decision" (i.e. choosing the right database product).

2.11 Utilities

Some utilities can be provided by the vendor and some by the third party.

Examples of typically needed utilities:

  • Load routines
  • Unload/reload and dump/restore routines
  • Reorganization routines
  • Statistical routines

3. An Introduction to Relational Databases

3.2 An Informal Look at the Relational Model

Three main aspects:

  • Structural aspect: The data in the database is perceived by the user as tables, and nothing but tables.
  • Integrity aspect: Those tables satisfy certain integrity constraints, to be discussed toward the end of this section.
  • Manipulative aspect: three most important are
    • restrict - extracts specified rows from a table,
    • project - extract specified colums from a table,
    • join - combines two tables into one on the basis of common values in a common column.

The result of each of the three operations (restrict, project, join) is another table, and thus the output from one operation can become input to another, i.e.

it is possible to write nested relational expressions.

The system tries very hard not to materialize intermediate results in their entirety:

  • materialized evaluation - intermediate results are fully materialized,
  • pipelined evaluation - intermediate results are passed piecemeal to subsequent operations.

The operations are all set-at-a-time, not row-at-a-time, i.e. the operands and results are whole tables, not just single rows, and tables contain sets of rows. By contrast, the operations in non-relational systems are typically at the row- or record-at-a-time level; thus, this set processing capability is a major distinguishing characteristis of relational systems.

The conceptual and external levels (logical structure) in a relational system will both be relational, but the internal level will not be.

The information principle:

  • the entire information content of the database is represented in one and only one way - as explicit values in column positions in rows and tables.

There are no pointers connecting one table to another at the conceptual and external levels (i.e. seen by the user), as in non-relational database systems.

A more formal definition

Five components of the relational model:

  1. An open-ended collection of scalar types (including the particular the type of boolean).
  2. A relation type generator and an intended interpretation for relations of types generated thereby.
  3. Facilities for defining relation variables of such generated relation types.
  4. A relation assignment operation for assigning relation values to such relation variables.
  5. An open-ended collection of generic relational operators ("the relational algebra") for deriving relation values from other relation values.

3.3 Relations and relvars

Relation is a mathematical term for the table of a special kind (see Ch. 6 for details):

  • the relational model is mainly based on set theory and predicate logic,
  • Codd E.F "A Relational Model of Data for Large Shared Data Banks"

Now, suppose we implement DELETE operation in SQL - we're deleting row number 3 in the given table table_1.

DELETE table_1 WHERE row = row (3);

In this case the old relation value of table_1 is replaced en bloc by an entirely new relation value which looks very similar (they differ with one row only), but conceptually they are different values (see Ch. 6). We have an assignment operation:

table_1 := table_1 WHERE NOT (row = row (3) );
  • the expression on the right side is evaluated,
  • the result of that evaluation is assigned to the variable on the left side.

There must be the distinction between relation values and relation variables.

Note: update operations and integrity constraints (see Ch. 6 and 9 respectively) both apply specifically to relvars, not relations.

3.4 What Relations Mean

A very important way of thinking about relations:

  1. Given a relation r, the heading of r denotes a certain predicate (where a predicate is just a truth-valued function that takes a set of paremeters).
  2. Each row in the body of r denotes a certain true proposition, obtained from the predicate by substituting certain argument values of the appropriate type for the parameters of the predicate ("instantiating the predicate").

These headings are types (domains). Types are to relations as nouns are to sentences.

  1. Types and relations (relation value) are both necessary (without types we have nothing to talk about, and without relations we cannot say anything).
  2. Types and relations are sufficient, as well as necessary - that is, we do need anything else, logically speaking.
  3. Types and relations are not the same thing.

Every relation has an associated predicate, including relations that are derived from others by means of operations such as join.

Relvars have predicates too: the predicate that is common to all of the relations that are possible values of the relvar in question.

3.7 Base relvars and views

Base relvars - original relvars, and base relations are their values:

  • CREATE TABLE relname, where TABLE is the the base relvar with the name relname.

A relation that can be obtained from the base relations is called derived, or derivable, relation, and the relvar is called view, or virtual relvars:

  • CREATE VIEW viewname AS ..., where
    • VIEW is view,
    • ... is view-defining expression which is not evaluated but remembered by the system in the catalog, but the user can call it as a regular relvar in the database with the name viewname.

Expressible relations are collectively base and derived relations.

View is not the copy of the data, it is just another view of base relations, so any changes to views will be applied to the base relations. When you query the view, the query is optimized and executed on the base relvars, so for the human the view is just like a shortcut (kind of function?).

A view definition can be of arbitrary complexity (it can even refer to other views).

The fact that the view is a relvar is precisely one of the strengths of relational systems (like subset is also a set, in mathematics). Thus, tables are base relvars and views.

And one should never forget that the internal, conceptual, and external views of ANSI/SPARK are just an abstraction on the physically stored data, and the base relvars should not be considered as "real" data representation.

4.6 Embedded SQL

SQL statements can be executed:

  • directly - interactively from an online terminal,
  • as part of an application program - i.e. embedded into another language.

A fundamental principle is dual-mode principle:

  • any SQL statement that can be used interactively can also be embedded in an application program.

But several embedded SQL statements cannot be used interactively.

Retrieval operations require special treatment: set-level retrieval capabilities of SQL in general are not supported by host languages. Such SQL operations retrieve many rows at a time and the host language can handle only one row at a time, so we need a cursor.

The cursor is a logical pointer in the application that can be used to run through a collection of rows.

Operations Not Involving Cursors

  • Singleton SELECT - expression that evaluates to a table containing at most one row,
  • INSERT,
  • DELETE (except the CURRENT form),
  • UPDATE (except the CURRENT form).

Operations Involving Cursors

Retrieval of a set containing an arbitrary number of rows:

DECLARE mycursor CURSOR FOR SELECT ...

This statement is declarative. A program can include any number of of DECLARE CURSOR statements.

Three executable statements are provided to operate on cursors:

  • OPEN - a set of rows is identified and becomes the current active set for the cursor; the cursor identifies the a position within that active set, namely, the position just before the first row; active set are always considered to have an ordering - explicit (ORDER BY) or system, so they are list rather than sets (sets do not have order in programming languages);
  • FETCH - advances the cursor to the next row in the active set; if there is no next row, SQLSTATE is set to 02000 and no data is retrieved.
  • CLOSE - closes the cursor.
  • CURRENT forms of UPDATE and DELETE - execute a command on the current position of the cursor.

4.7 Dynamic SQL and SQL/CLI

Steps that the app with embedded SQL does:

  1. Accept a command from the terminal.
  2. Analyze the command.
  3. Execute appropriate SQL statements on the database.
  4. Return a message and/or results to the terminal.

Dynamic SQL

Dynamic SQL is part of embedded SQL. It consists of a set of "dynamic statements" - wich themselves are compiled ahead of time - whose purpose is precisely to support the compilation and execution of regular SQL statements that are constructed at runtime.

Two principal dynamic statements are:

  • PREPARE,
  • EXECUTE.

Example from PostgreSQL:

PREPARE fooplan (int, text, bool, numeric) AS
    INSERT INTO foo VALUES($1, $2, $3, $4);
EXECUTE fooplan(1, 'Hunter Valley', 't', 200.00);

The SQL standard includes a PREPARE and EXECUTE statements, but it is only for use in embedded SQL. These versions of the PREPARE and EXECUTE statements also use a somewhat different syntax.

Call-Level Interface

SQL/CLI feature was added to the standard in 1995 and is heavily based on Microsoft's Open database connectivity interface, ODBC.

  • Dynamic (embedded) SQL is a source code standard, and an application needs the services of some kind of SQL compiler in order to process the operations, like PREPARE and EXECUTE. SQL/CLI does not need any special compiler, only the regular services of the standard host languages compiler, thus this applications can be distributes in object code form.

  • Such applications, then, can be DBMS-independent.

    • Performance Optimization: For certain operations, especially those involving large datasets or complex computations, performing the processing within the database using PL/Python can be more efficient than fetching data to the client application for processing.

  • Use the psycopg library when:

    • Client-Side Processing: If you need to retrieve data from the database and perform extensive processing, analysis, or visualization in your Python application outside the database environment, psycopg is the better choice.     • Data Retrieval and Manipulation: psycopg is suitable for tasks such as querying the database, fetching results, updating records, and performing any data manipulation that doesn't require the computational resources of the database server.

In summary, use PL/Python when you need to perform intensive data processing within the database environment itself, and use psycopg when you need to interact with the database from your Python application for tasks such as data retrieval, manipulation, and application development. The "part-explosion problem" in relational databases refers to a situation where the number of rows returned by a query grows exponentially due to the presence of one-to-many or many-to-many relationships between tables.

This problem typically arises when performing joins across tables that have a one-to-many relationship and selecting multiple columns from these tables. Each row in the "one" table may correspond to multiple rows in the "many" table, leading to a proliferation of rows in the result set.

For example, consider a database schema with two tables: Order and OrderItem. Each order can have multiple order items. If you join these tables and select columns from both, you may end up with multiple rows for each order, each representing a different order item. If an order has, say, three order items, the query result will contain three rows for that order, leading to duplication of order-level information.

The part-explosion problem can make query results larger than expected and can impact performance, especially when dealing with large datasets. It often requires careful consideration of query design, including the use of aggregation functions, grouping, or filtering to mitigate the problem.

To address the part-explosion problem, database designers and developers can employ techniques such as normalization (to reduce redundant data), carefully crafted joins, use of subqueries or common table expressions, and appropriate indexing to optimize query performance and minimize unnecessary data duplication.
